# Feedback
Corrections and confirmations from the user. The agent writes here when:
- User pushes back on an approach ("no, do X", "stop doing Y").
- User confirms a non-obvious choice ("yes exactly", accepting an unusual call without pushback).

## File-per-topic
Name files by topic, not date: `feedback_brevity.md`, `feedback_routing.md`, etc.

## Required structure
```
# <topic>
Status: active

<the rule itself in one sentence>

**Why:** <reason the user gave — often a past incident or strong preference>
**How to apply:** <when/where this kicks in>
```

## Why "why" matters
Knowing the reason lets the agent judge edge cases instead of blindly following the rule.
A rule without a reason rots the moment context shifts.
